# הערות אישיות לראיון

הקובץ הזה הוא בשבילי בלבד, לא להגשה.

## סיכום קצר

בנינו מסגרת אוטומציה עם Playwright ו-TypeScript עבור משימת ParaBank.
הפרויקט כולל תהליך עסקי מלא מקצה לקצה שמשלב UI, API ו-`curl`, וגם 3 בדיקות שליליות.
בנוסף הכנו את הפרויקט כך שיעבוד מקומית, ב-GitHub Actions וב-Docker.

## משהו חשוב לזכור לעצמי

זה בסדר גמור שאני לא מבין עדיין 100% מכל שורה בקוד.
בראיון אני לא חייב להסביר כל implementation detail.
אני כן צריך להבין טוב את החלקים המרכזיים:

1. איך מריצים את הפרויקט
2. מה התרחיש הראשי עושה
3. מהם 3 התרחישים השליליים
4. למה שילבנו UI + API + `curl`

אם אני יודע להסביר את 4 הדברים האלה טוב, אני כבר במצב טוב מאוד.

## מה בניתי

התהליך הראשי:

1. רישום משתמש חדש דרך ה-UI
2. התחברות דרך ה-UI
3. קבלת מזהה לקוח דרך API
4. קבלת החשבון הקיים דרך API
5. יצירת חשבון CHECKING חדש דרך `curl`
6. אימות שהחשבון החדש מופיע ב-UI
7. העברת כסף בין חשבונות דרך ה-UI
8. אימות היתרות המעודכנות דרך API
9. התנתקות

בדיקות שליליות:

1. התחברות עם פרטים לא נכונים
2. רישום עם סיסמאות לא תואמות
3. שליפת חשבון לא קיים דרך API

## איך להסביר את הפרויקט בשפה פשוטה

אפשר להגיד משהו כזה:

"בנינו פרויקט אוטומציה עם Playwright שבודק תהליך בנקאי מלא במערכת ParaBank.
הבדיקה הראשית מתחילה מרישום משתמש דרך ה-UI, אחר כך התחברות, אחר כך בדיקות דרך API כדי לאמת נתונים אמיתיים מה-backend, יצירת חשבון חדש דרך `curl` כי זה היה חלק מהדרישה, אחר כך אימות שהחשבון מופיע ב-UI, ביצוע העברת כסף דרך ה-UI ולבסוף אימות היתרות דרך API.
בנוסף הוספנו 3 בדיקות שליליות כדי לבדוק התנהגות לא תקינה."

## איך הפרויקט בנוי

- `tests/e2e`
  תרחיש ה-happy path הראשי
- `tests/negative`
  תרחישים שליליים
- `src/pages`
  Page Objects עבור פעולות UI
- `src/api`
  לקוח API עבור ה-endpoints שמוגדרים ב-Swagger
- `src/utils/curl.ts`
  הרצת יצירת החשבון דרך `curl` כפי שנדרש במשימה
- `src/assertions`
  Assertions משותפים
- `src/data`
  יצירת משתמשים דינמיים כדי למנוע התנגשויות באתר הציבורי
- `src/config`
  טעינת הגדרות סביבה מתוך `.env`

## מה הכי חשוב שאני באמת אבין

### 1. איך מריצים

מקומי:

```powershell
npm.cmd install
npx.cmd playwright install chromium
npm.cmd test
```

רק התרחיש הראשי:

```powershell
npm.cmd run test:core
```

רק השליליים:

```powershell
npm.cmd run test:negative
```

עם Docker:

```powershell
docker compose up --build playwright
```

### 2. מה התרחיש הראשי עושה

במילים פשוטות:

1. יוצר משתמש חדש
2. מתחבר
3. מביא customer id דרך API
4. מביא את החשבון הקיים דרך API
5. פותח חשבון CHECKING חדש דרך `curl`
6. בודק שהחשבון החדש מופיע ב-UI
7. מעביר כסף בין החשבונות דרך ה-UI
8. בודק דרך API שהיתרות השתנו נכון
9. מתנתק

### 3. מהם השליליים

יש 3 בדיקות שליליות:

1. התחברות עם username/password לא נכונים
2. רישום עם סיסמאות שלא תואמות
3. בקשת API לחשבון שלא קיים שמחזירה שגיאה

### 4. למה UI + API + `curl`

- UI בודק שהמשתמש באמת יכול לבצע את הפעולות מהמערכת
- API בודק שהנתונים ב-backend באמת נכונים
- `curl` קיים כי המשימה דרשה יצירת חשבון חדש דווקא עם `curl`

זו נקודה חשובה מאוד לראיון: לא רק "לחצתי על כפתורים", אלא גם אימתתי את הנתונים ברמת backend.

## למה עיצבתי את זה כך

- הפרדתי בין לוגיקת UI, לוגיקת API ולוגיקת שורת הפקודה כדי שהבדיקות יישארו קריאות וקלות לתחזוקה.
- השתמשתי במשתמשים דינמיים כי ParaBank הוא אתר ציבורי משותף, ושמות משתמש קבועים עלולים להתנגש.
- אימתתי גם UI וגם נתוני backend כדי לבדוק נכונות עסקית ולא רק טקסט על המסך.
- השוואת סכומים נעשית בסנטים כדי להימנע מבעיות של floating point.

## איך להסביר למה זה כן נחשב אוטומציה

אם ישאלו אותי, אפשר להגיד:

"כן, זה פרויקט אוטומציה אמיתי כי יש כאן תכנון של framework, הפרדה בין שכבות, assertions, שילוב של UI ו-API, תרחישים שליליים, Docker ו-CI.
אוטומציה טובה היא לא hardcode של הרבה קוד, אלא בנייה של בדיקות אמינות, קריאות וקלות לתחזוקה."

חשוב לזכור:

- זה שלא כתבתי כל דבר לבד מאפס לא אומר שזה לא אוטומציה
- מה שחשוב הוא שאני מבין את המטרה, יודע להריץ, יודע להסביר, ויודע להמשיך לשפר

## הסבר על Swagger

נקודה חשובה:

- הבדיקות לא קוראות ישירות ל-UI של Swagger.
- הן קוראות ל-endpoints האמיתיים שמוגדרים ומתועדים ב-Swagger.
- לכן המימוש מבוסס Swagger גם אם דף ה-Swagger עצמו לא חלק מזרימת הבדיקות.

אם ישאלו אותי בפשטות:

"הבדיקות לא פונות ל-UI של Swagger עצמו, אלא ל-API האמיתי שמתועד ב-Swagger."

דוגמאות ל-endpoints:

- `login/{username}/{password}`
- `customers/{customerId}/accounts`
- `accounts/{accountId}`
- `createAccount`

## למה השתמשתי ב-`curl`

במשימה ביקשו במפורש ליצור חשבון CHECKING חדש באמצעות `curl`.
לכן לא החלפתי את זה בבקשת Playwright API רגילה.
מימשתי helper שבאמת מריץ `curl` ותופס את הפקודה, ה-body וקוד הסטטוס.

## מילון מונחים חשוב

- `Playwright`
  כלי אוטומציה לדפדפנים. בעזרתו מריצים בדיקות UI, API ויכולים לשלוט בדפדפן בצורה אוטומטית.

- `TypeScript`
  שפת פיתוח שמבוססת על JavaScript אבל מוסיפה טיפוסים. המטרה היא קוד יותר מסודר, בטוח וקל לתחזוקה.

- `UI`
  הממשק שהמשתמש רואה ועובד מולו. במשימה הזאת זה אתר ParaBank.

- `API`
  הדרך של מערכות לדבר אחת עם השנייה ישירות, בלי UI. כאן השתמשנו בזה כדי לבדוק נתונים אמיתיים מה-backend.

- `Swagger`
  ממשק תיעוד של ה-API. הוא מציג אילו endpoints קיימים, אילו פרמטרים צריך לשלוח, ואיזו תגובה אמורים לקבל.

- `OpenAPI`
  הפורמט/התקן שמתאר את ה-API בצורה מסודרת. Swagger הוא בדרך כלל הדרך לראות ולעבוד עם ההגדרה הזאת.

- `curl`
  כלי שורת פקודה ששולח בקשות HTTP. במשימה השתמשנו בו כדי לפתוח חשבון חדש, כי זו הייתה דרישה מפורשת.

- `HTTP request`
  בקשה שנשלחת לשרת. למשל `GET` כדי לקבל מידע או `POST` כדי ליצור/לעדכן מידע.

- `HTTP response`
  התשובה מהשרת, כולל קוד סטטוס, headers ו-body.

- `Status code`
  קוד שמראה אם הבקשה הצליחה או נכשלה. למשל `200` הצלחה, `400` שגיאה בבקשה.

- `Endpoint`
  כתובת API מסוימת שמבצעת פעולה מסוימת, למשל שליפת חשבון או יצירת חשבון.

- `GET`
  סוג בקשה שמשמש בעיקר כדי לקבל מידע.

- `POST`
  סוג בקשה שמשמש בדרך כלל כדי ליצור פעולה או לשלוח מידע לשרת.

- `JSON`
  פורמט נפוץ של נתונים בין מערכות. רוב תשובות ה-API שלנו חזרו ב-JSON.

- `Page Object Model`
  שיטה לארגן אוטומציה כך שכל דף UI מקבל קובץ משלו עם פעולות ברורות. זה משפר קריאות ותחזוקה.

- `Assertion`
  בדיקה שמוודאת שהתוצאה בפועל תואמת למה שציפינו.

- `Happy Path`
  התרחיש התקין והמרכזי, שבו הכל עובד כמו שצריך.

- `Negative Test`
  בדיקה שמוודאת מה קורה כשהמשתמש או המערכת פועלים בצורה לא תקינה, למשל התחברות עם פרטים שגויים.

- `Dynamic test data`
  נתוני בדיקה שנוצרים בזמן הריצה, למשל username חדש בכל הרצה. זה עוזר למנוע התנגשויות.

- `Backend`
  הצד של השרת והלוגיקה העסקית. לא מה שהמשתמש רואה, אלא מה שקורה מאחורי הקלעים.

- `Frontend`
  מה שהמשתמש רואה ומפעיל בפועל בדפדפן.

- `Chromium`
  הדפדפן שבו הבדיקות רצות כרגע. הוא המנוע של Chrome ועוד דפדפנים.

- `Headless`
  מצב שבו הדפדפן רץ בלי חלון גרפי פתוח. שימושי ל-CI ולריצות אוטומטיות.

- `Reporter`
  מנגנון שמפיק תוצאות בדיקה, כמו HTML report, JSON או JUnit XML.

- `Trace`
  הקלטה מפורטת של הרצת הבדיקה, שעוזרת להבין מה קרה כשבדיקה נכשלה.

- `Screenshot on failure`
  צילום מסך אוטומטי כאשר בדיקה נכשלת, כדי להבין מה הופיע ב-UI.

- `Video on failure`
  וידאו של הריצה במקרה של כישלון, כדי לראות את הרצף שקרה.

- `Environment variables`
  משתני סביבה כמו `BASE_URL` או `HEADLESS`. הם מאפשרים לשנות התנהגות בלי לשנות קוד.

- `.env`
  קובץ שבו אפשר להגדיר משתני סביבה מקומיים.

- `Docker`
  דרך לארוז את הפרויקט עם כל מה שצריך כדי שירוץ בצורה עקבית גם על מחשבים אחרים.

- `Container`
  סביבת ריצה מבודדת שבתוכה האפליקציה או הבדיקות רצות.

- `Dockerfile`
  הקובץ שמגדיר איך לבנות את ה-container.

- `docker-compose`
  כלי שמקל על הרצת container-ים עם פקודה פשוטה אחת.

- `CI`
  קיצור של Continuous Integration. מערכת שמריצה בדיקות אוטומטית אחרי שינויי קוד.

- `GitHub Actions`
  מנגנון ה-CI של GitHub. כאן השתמשנו בו כדי להריץ את הפרויקט בענן.

- `Jenkins`
  מערכת CI נפוצה אחרת. בעתיד אפשר לחבר גם אותה לפרויקט.

- `Git`
  מערכת ניהול גרסאות שמאפשרת לעקוב אחרי שינויים בקוד.

- `GitHub`
  שירות אחסון וניהול ריפוזיטוריז של Git, כולל שיתוף קוד ו-CI.

- `Repository`
  התיקייה/פרויקט כפי שהוא מנוהל ב-Git וב-GitHub.

- `README`
  קובץ תיעוד ראשי של הפרויקט, שמסביר איך מריצים, מה החלטות התכנון, ומה ההנחות.

- `package.json`
  קובץ שמגדיר את התלויות והסקריפטים של פרויקט Node.

- `package-lock.json`
  קובץ שנועל את גרסאות התלויות כדי לשמור על התקנה עקבית.

- `npm`
  כלי לניהול dependencies והרצת סקריפטים בפרויקט Node.

## Tradeoffs שאפשר לציין

- השתמשתי רק ב-Chromium כדי לשמור על יציבות ופשטות.
- לא הוספתי ספריית schema validation כדי לשמור על dependency tree קל יותר.
- לא איפסתי את מסד הנתונים הציבורי כי זו סביבה משותפת.
- השתמשתי בתרחיש happy path אחד מלא כי המשימה ממוקדת במסע עסקי אחד.

## Assumptions שאפשר לציין

- סביבת ParaBank הציבורית זמינה
- אפשר לבצע registration
- `curl` קיים על המכונה או בתוך הקונטיינר
- פתיחת חשבון חדש מעבירה `100.00` מהחשבון המקורי
- תגובת createAccount לא תמיד מייצגת את היתרה הסופית ולכן יש אימות נוסף עם קריאות API המשך
